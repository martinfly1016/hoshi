命式计算库评测

用途：记录计算库选型、人工对照工具结果、样例输入输出和最终判断。这个页面只服务内部评测，不承接线上视觉规范。

当前阶段：资料初筛 + 待 Spike 首选候选：tyme4ts 对照候选：lunar 系列 人工对照：问真八字 / 八字测测专业版
计算层目标：只输出可信、可复核的结构化命式 JSON。解释文案由后续其它模型完成，不进入计算库选型范围。
tyme4ts 首选 Spike。TypeScript 友好，MIT，需验证八字 API 完整度。
lunar 系列 成熟基准。适合作为 tyme4ts 的对照和备选。
问真八字 主人工对照。用于验证四柱、节气、大运、流年。
测测 辅助参考。更适合看报告结构和解释体验。

1. 候选库对比

候选方案 定位 优势 风险 / 劣势 当前判断
tyme4ts 首选底层历法库 TypeScript 友好；MIT；依赖面小；与 6tail lunar 系列同源。 需要确认 API 是否能直接输出完整四柱，以及节气边界是否满足要求。 优先 Spike
lunar-typescript / lunar-javascript 成熟基准与备选 使用资料更多；八字能力更直接；适合做对照验证。 若使用 JS 版本，需要做 TS 类型封装；也需要验证边界规则。 同步验证
lunisolar 参考实现 功能覆盖强，适合参考能力边界和 API 设计。 GPL-3.0 许可证对商业闭源产品有风险。 不建议生产依赖
bazi-calculator-by-alvamind 直接八字分析库 概念接近需求，可能有现成分析结构。 成熟度、维护状态、边界样例可信度都需要重验。 仅参考
dantalion 生日/占卜应用库 可参考产品包装方式。 不适合作为四柱命式计算核心。 不采用
完全自研 长期可控方案 规则完全可控，便于做流派差异。 MVP 阶段校验成本过高，容易在节气和边界规则出错。 暂缓

4. 输出结构约束

无论最终使用哪个库，产品层只依赖统一函数，不直接依赖第三方 API。

层级 职责 说明
计算层 calculateShichusuimei(input) 输出四柱、藏干、五行、十神、大运、流年基础结构。
Adapter 层 adapters/tyme、adapters/lunar 隔离第三方库 API，便于替换或双跑验证。
解释输入层 结构化 JSON 转模型上下文 只做字段编排，不在这里写命理解释。
解释模型 生成最终解读 后续可替换模型或做不同解释风格。
