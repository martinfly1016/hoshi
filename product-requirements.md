# 星の命式 产品需求文档

版本: v0.1  
日期: 2026-05-09  
市场: 日本  
产品方向: 日本向け占い・玄学工具站  
建议品牌: 星の命式 / Hoshi no Meishiki  
建议主域名: `hoshinomeishiki.jp`

## 1. 背景与结论

基于现有 SEO 调研，日本市场中“四柱推命”具备最高的初期投入价值：搜索量足够大，用户认知成熟，且可以自然延展到“免费鉴定”“相性占い”“生年月日占い”等高意图页面。

本产品不建议以泛占卜门户作为第一期形态，也不建议直接移植中文语境下的“八字排盘”。日本用户更熟悉“四柱推命”“命式”“生年月日占い”“相性占い”等表达，因此第一期应以日本用户的搜索语言和理解路径为主。

产品方案确定为：

- 第一阶段聚焦“四柱推命 免费鉴定工具”。
- 同步建设“四柱推命说明页”“相性诊断入口”“生年月日占い入口”。
- 紫微斗数作为第二阶段方向，第一期只做轻量预埋或栏目占位。
- 奇门遁甲不进入 MVP。

## 2. 产品定位

星の命式是面向日本用户的命式类占卜工具站，第一期通过四柱推命帮助用户根据出生信息生成命式，并用易懂的日文解释性格、恋爱、工作、五行平衡和未来运势入口。

产品核心不是“神秘感页面”，而是“可理解、可保存、可分享的命式结果工具”。

### 2.1 核心价值

- 用户无需理解专业术语，也能看懂自己的命式特点。
- 用户可以通过生年月日、出生时间、出生地生成基础命式。
- 结果页同时满足工具体验和 SEO 内容消费。
- 支持后续扩展到相性、紫微斗数、流年、大运等模块。

### 2.2 品牌原则

- 使用日本用户能理解的术语：四柱推命、命式、生年月日占い、相性占い。
- 避免关键词堆砌型站名和廉价占卜站质感。
- 视觉和文案应体现“清晰、可信、温和”，而不是夸张玄学营销。

## 3. 目标用户

### 3.1 主用户

日本 20-45 岁，对占卜、性格分析、恋爱关系、职业方向有兴趣的普通用户。

典型入口搜索词：

- 四柱推命
- 四柱推命 無料
- 四柱推命 無料 当たる
- 四柱推命 相性
- 生年月日 占い

用户特点：

- 不一定知道四柱推命的详细体系。
- 更关心结果是否好懂、是否像“说中了自己”。
- 手机访问占比预计较高。
- 对复杂表格和术语有理解门槛。

### 3.2 次用户

对命理已有一定了解的用户，可能会搜索命式、通变星、五行、大運、流年等关键词。

这类用户对准确性和信息完整性更敏感，需要在结果页提供专业信息展开入口。

## 4. 产品目标

### 4.1 MVP 目标

上线一个围绕“四柱推命 免费鉴定”的可用工具站，完成从搜索进入、填写出生信息、生成命式、阅读解释、继续访问相性或扩展内容的闭环。

### 4.2 业务目标

- 建立 `hoshinomeishiki.jp` 的日本占卜工具品牌基础。
- 抢占“四柱推命 免费”“四柱推命 相性”“生年月日占い”等 SEO 页面。
- 为后续紫微斗数、相性诊断、流年运势等功能建立可复用框架。

### 4.3 成功指标

MVP 上线后重点观察：

- 自然搜索收录页面数。
- `/shichusuimei/free/` 的访问量和输入完成率。
- 结果页停留时间。
- 结果页到相性诊断入口的点击率。
- 移动端表单完成率。
- Search Console 中四柱推命相关关键词曝光增长。

## 5. MVP 范围

### 5.1 必须包含

#### 首页 `/`

首页承担品牌介绍和主要工具分发，不做复杂门户。

需要包含：

- 品牌名：星の命式。
- 主入口：四柱推命 無料鑑定。
- 次入口：四柱推命 相性、生年月日占い。
- 紫微斗数作为“準備中”或栏目预告。
- 简短说明本站可以做什么。

#### 四柱推命主页面 `/shichusuimei/`

面向“四柱推命”主关键词。

需要包含：

- 四柱推命是什么。
- 命式中年柱、月柱、日柱、时柱的基础说明。
- 五行、通变星、十二运等基础概念入口。
- 免费鉴定工具入口。
- 相性诊断入口。
- SEO 向 FAQ。

#### 四柱推命免费鉴定 `/shichusuimei/free/`

MVP 的核心工具页面。

输入项：

- 生年月日。
- 出生时间，可选择“不明”。
- 出生地，至少支持日本都道府县；后续可扩展城市和海外。
- 性别可选，但不应作为基础命式必填项，除非用于大运顺逆等计算。

输出项：

- 命式表。
- 日主。
- 五行平衡图。
- 通变星摘要。
- 性格倾向。
- 恋爱倾向。
- 工作/才能倾向。
- 当前结果的注意说明。
- 大运、流年、相性诊断的后续入口。

交互要求：

- 移动端优先。
- 表单步骤清晰，输入成本低。
- 结果先给摘要，再给专业表格。
- 术语可展开解释。
- 结果可保存或分享为图片，该项可以作为 MVP 后半段功能，若工期紧可延后。

#### 相性诊断入口 `/shichusuimei/compatibility/`

第一期可以做轻量版，不必须完成复杂合盘。

MVP 最小形态：

- 双人生年月日输入。
- 展示双方基础日主、五行倾向、关系摘要。
- 提示完整相性分析后续开放。

如果计算引擎尚未稳定，允许第一期先作为 SEO + 等待列表/入口页，但页面结构必须为后续工具化预留。

#### 生年月日占い `/birthday-fortune/`

作为非专业用户入口。

页面策略：

- 使用“生年月日占い”承接泛用户搜索。
- 解释其结果基于四柱推命的一部分。
- 引导用户进入免费命式鉴定。

### 5.2 第二阶段预埋

#### 紫微斗数

建议 URL：

- `/shiheitousu/`
- `/shiheitousu/free/`
- `/shiheitousu/chart/`

### 5.3 暂不包含

MVP 不做：

- 奇门遁甲工具。
- 付费咨询撮合。

- AI 聊天式占卜。
- 多语言版本。
- 完整紫微斗数命盘。

## 7. 核心用户流程

### 7.1 免费鉴定流程

1. 用户通过搜索进入 `/shichusuimei/free/`。
2. 用户填写出生日期、出生时间、出生地。
3. 系统生成命式。
4. 用户先看到一句话总结和关键标签。
5. 用户继续查看五行图、命式表、性格、恋爱、工作解释。
6. 用户点击相性诊断、大运、流年或保存分享结果。

### 7.2 生年月日占い流程

1. 用户搜索“生年月日 占い”进入 `/birthday-fortune/`。
2. 页面用更轻量语言解释可通过出生信息查看性格和运势。
3. 用户进入四柱推命免费鉴定。
4. 系统在结果页提供更完整解释。

### 7.3 相性流程

1. 用户从 `/shichusuimei/` 或结果页进入 `/shichusuimei/compatibility/`。
2. 输入两个人的出生信息。
3. 获取基础相性摘要。
4. 引导查看双方命式或后续完整相性报告。

## 8. 功能需求

### 8.1 出生信息输入

字段要求：

| 字段 | 必填 | 说明 |
|---|---|---|
| 生年月日 | 是 | 年、月、日 |
| 出生时间 | 否 | 支持“不明” |
| 出生地 | 是 | MVP 至少支持日本都道府县 |
| 性别 | 否 | 若用于大运顺逆再启用 |

校验要求：

- 日期不能晚于当前日期。
- 出生时间未知时，结果需明确提示准确性限制。
- 出生地无法解析时，允许使用默认时区 `Asia/Tokyo`，但需在结果中说明。

### 8.2 四柱推命计算

本产品不能使用原型中的伪逻辑。本番化必须使用可靠计算方案。

计算需求：

- 年柱、月柱、日柱、时柱。
- 天干地支。
- 五行归属。
- 通变星。
- 藏干。
- 十二运。
- 大运，至少预留接口。
- 流年，至少预留接口。

准确性要求：

- 需要处理节气切换，不能简单按公历月份划分月柱。
- 需要明确出生地、时区、真太阳时是否纳入计算。
- MVP 可以先采用日本国内出生、`Asia/Tokyo` 时区作为默认范围，但文档和 UI 需要说明。
- 若不支持真太阳时，应在结果中标注“出生地による補正は簡易計算です”等提示。

### 8.3 结果展示

结果页展示顺序：

1. 一句话摘要。
2. 用户基础信息确认。
3. 关键标签：日主、五行偏向、主要通变星。
4. 五行平衡图。
5. 命式表。
6. 性格解释。
7. 恋爱解释。
8. 工作/才能解释。
9. 专业术语展开区。
10. 后续入口：相性、大运、流年。

文案原则：

- 避免绝对化断言。
- 避免制造焦虑。
- 使用“傾向があります”“可能性があります”等柔和表达。
- 专业词汇先解释，再展开。

### 8.4 SEO 内容模块

每个工具页都应包含自然衔接的 SEO 内容，而不是工具和文章割裂。

`/shichusuimei/free/` 需要覆盖：

- 四柱推命無料鑑定とは。
- 生年月日と出生時間が必要な理由。
- 命式でわかること。
- 出生時間が不明な場合。
- 結果の見方。
- FAQ。

`/shichusuimei/compatibility/` 需要覆盖：

- 四柱推命の相性とは。
- 日主与五行在相性中的作用。
- 恋愛相性、仕事相性、結婚相性。
- FAQ。

## 9. 非功能需求

### 9.1 移动端体验

- 表单控件必须适合手机输入。
- 结果页不能以宽表格作为第一屏核心内容。
- 命式表在手机上需要横向滚动或重排。
- 图表和摘要需要优先展示。

### 9.2 性能

- 初期页面应尽量静态化或 SSR。
- 工具计算应在服务端或可信前端模块中完成。
- 首屏加载应控制在轻量范围内，避免过重动画。

### 9.3 可扩展性

计算引擎、解释规则和页面展示应分离：

- 计算层：输入出生信息，输出结构化命式。
- 解释层：根据结构化命式生成摘要和分类解释。
- 展示层：负责表格、图表、页面布局。

这样后续可以扩展紫微斗数、相性、流年和保存分享功能。

### 9.4 合规与信任

- 页面需声明结果仅供娱乐和自我理解参考。
- 不提供医疗、投资、法律等高风险建议。
- 对“出生时间不明”“简易计算”“真太阳时未补正”等情况明确提示。
- 若未来收集用户出生信息，需要隐私政策和数据保留策略。

## 12. 技术方案建议

### 12.1 推荐架构

建议使用适合 SEO 的前端框架和静态/服务端渲染能力，例如 Next.js 或 Astro。

推荐理由：

- 工具页需要 SEO 收录。
- 内容页和工具页可以共存。
- 后续可扩展 API、图像分享、用户结果保存。

### 12.2 计算引擎原则

计算引擎是本产品的风险核心。第一期需要优先确认：

- 是否使用可靠开源库。
- 是否自行实现节气与干支计算。
- 是否支持日本时区和出生地。
- 如何验证计算结果。

验收方式：

- 准备至少 20 组已知命例。
- 与可信四柱推命排盘工具交叉比对。
- 对节气切换日前后样例重点测试。
- 对出生时间未知和时辰边界样例重点测试。

## 13. MVP 验收标准

MVP 可上线标准：

- 用户可以在手机上完成出生信息输入。
- 能生成结构化命式，且基础干支计算通过样例验证。
- 结果页有摘要、五行图、命式表和至少三类解释。
- `/shichusuimei/free/` 有完整 SEO 内容和 FAQ。
- 页面有免责声明和计算限制说明。
- Search Console、基础访问统计可用。
- 主要页面具备 title、description、canonical、OG 信息。

## 14. 风险与对策

| 风险 | 影响 | 对策 |
|---|---|---|
| 四柱推命计算不准确 | 影响信任和口碑 | 先确定计算引擎，建立样例测试 |
| 内容过浅 | 无法建立专业感 | 工具结果中提供术语展开和内容页内链 |
| 移动端表单复杂 | 输入完成率低 | 分步输入、默认日本时区、出生时间可未知 |
| SEO 页面薄 | 收录和排名弱 | 工具页结合说明、FAQ、结果示例 |
| 品牌过窄 | 后续难扩展 | 品牌使用“命式”，不绑定单一占术 |

## 15. 待确认问题

1. 是否立即购买 `hoshinomeishiki.jp` 和保护域名。
2. 新站是否独立仓库，还是从现有工程拆分。
3. 四柱推命计算采用开源库、自研，还是混合方案。
4. MVP 是否必须支持真太阳时。
5. 相性诊断第一期做完整功能还是入口页。
6. 结果分享图片是否进入第一期。
7. 是否需要保存用户出生信息；若需要，需要隐私策略。
